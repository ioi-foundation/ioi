Prompt 1 — Evidence Boundary / Parent-vs-Child Access Gauge

You are reconstructing the parent IOI harness/orchestrator, not merely the spawned child workspace or Codex child harness.

Your first task is not to write architecture. Your first task is to prove what layer you can actually see.

Produce docs/parent-harness-evidence-boundary.md.

Rules:

  • Do not infer parent behavior from child behavior unless clearly labeled as inference.
  • Do not call any claim “source-derived” unless you can cite a file path, command output, log line, schema, process, or source symbol.
  • If you only see the child workspace, say so clearly.
  • Never print secrets, tokens, auth headers, private keys, cookies, or full environment variables containing credentials. Redact sensitive values.
  • Prefer non-destructive inspection only.

Inspect and document:

  1. Current identity and runtime:

    • pwd
    • whoami
    • id
    • hostname
    • uname -a
    • current shell
    • container/runtime hints
    • IOI-specific environment variable names, with secret values redacted

  2. Workspace/repo status:

    • current git repo path
    • branch
    • remotes
    • commit history status
    • untracked files
    • whether this is an empty repo, generated repo, child checkout, or parent source checkout

  3. Process tree:

    • parent processes
    • agent processes
    • editor/server processes
    • IOI supervisor or workspace processes
    • any process that appears to have launched the child agent

  4. Mounted paths and accessible directories:

    • /workspace
    • /workspaces
    • /home/ioi
    • /tmp
    • /var/log
    • /var/lib
    • /ide
    • /opt
    • /app
    • any mounted volume paths visible from mount, df -h, or /proc/self/mountinfo

  5. Tooling boundary:

    • Which tools are child-agent tools?
    • Which tools might belong to the parent harness?
    • Which tools are only exposed through the conversation/runtime UI?
    • Which APIs/schemas are visible locally?

That's a direct question about the evidence boundary — here's what the inspection shows:

The clarifying question UI is part of the child harness.

Specifically:

  • The ask_clarifying_questions tool is listed in the agent's tool set, which runs entirely within ioi-swe-agent (the child harness binary at /usr/local/ioi/shared/ioi/ioi-swe-agent).
  • The agent communicates with the platform via the IOI_API_URL (https://app.ioi.io/api) and the LLM endpoint on the runner. The question/answer exchange travels through that API to reach the user's browser.
  • The conversation log is stored locally at /usr/local/ioi/shared/ioi/memory/conversations/019ed13a-5561-7c76-ac3c-9dabd511c75c.jsonl — the child harness writes it.

What this means for propagation:

The question is rendered in the IOI UI (which is served by the platform/parent), but the content originates from and is processed by the child agent. The parent harness acts as a transport and rendering layer — it receives the structured tool call output from the agent, displays it to the user, and routes the user's answer back as a message in the conversation. The parent does not interpret or act on the question independently; it passes the answer back to the child agent as the next conversation turn.

So: authored and processed in the child harness, transported and rendered by the parent harness. The parent has no semantic understanding of what was asked — it just moves the payload.

This is consistent with the ONA_AGENT_OFFLOAD_RESULTS=true env var observed in the evidence boundary: tool outputs above 5000 bytes are offloaded to durable storage, but the control flow stays in the child.
