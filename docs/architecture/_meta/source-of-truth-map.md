## Purpose

This map prevents split-brain architecture documentation. When a subject appears
in multiple plans, specs, evidence bundles, or implementation guides, edit the
canonical architecture file first and let supporting docs reference it.
Internal master guides are private execution scaffolding, not doctrine. They may
carry raw comparisons, sensitive sequencing, and transitional language; any
durable architectural decision they produce must be distilled into the canonical
owner below or into an accepted ADR before other docs or code treat it as canon.
After promotion, the internal guide is retired as an implementation authority:
builders should not chase it for target shape, readiness, route names, or
product taxonomy. Reverse-engineering evidence and local reference packets may
inform interaction shape, screenshots, and parity tests, but architecture
ownership, product names, object contracts, effect boundaries, and roadmaps live
in tracked `docs/architecture/`.

New readers should start with [`../START_HERE.md`](../START_HERE.md), which
points to the canonical [`start-here.md`](./start-here.md). Builders who already
know the stack can use [`implementation-matrix.md`](./implementation-matrix.md)
to map each concept to its durable form, owner doc, code anchor, and conformance
hook.

Validation anchors:

- Capability and authority refs are `prim:*` and `scope:*` families; legacy
  capability-prefix language is not canonical.
- Hypervisor App, Hypervisor Web, and Hypervisor CLI/headless are first-class
  clients over Hypervisor Core; Hypervisor Workbench is the live code/systems surface term.
  IDEs, browser shells, and external runners are adapter targets, not
  Hypervisor's product identity.
- IOI Authority Gateway is the daemon sidecar/compatibility profile. Do not
  claim the daemon authorizes anything; local/domain policy and the applicable
  authority provider authorize crossings. wallet.network is mandatory for
  portable delegated authority and designated high-risk external effects.
- Physical-action docs own `PhysicalActionPolicy` and `ActuatorCommandReceipt`.
- Runtime/API docs own SDK, CLI/headless, GUI, harness, benchmark, compositor, and agent-harness-adapter boundaries.
- Default harness docs own the Smarter-agent runtime loop.
- Architecture governance docs own Decision History Policy.
- aiagent broad-labor docs own `DigitalWorkerOntology`, `VerticalOntologyPack`,
  `IntegrationSurface`, and `ManagedWorkerInstance`.
- A GoalRun owns one bounded pursue/verify/course-correct loop. An
  `OutcomeRoom` / `CollaborativeWorkGraph` coordinates many GoalRuns; it is not
  a second runtime or an implicitly global Agentgres graph.
- `WorkResult` / `OutcomeDelta` is the generic result seam.
  `ImplementationResultPayload` is its software profile.
- Model route, Worker composition, runtime node, and independent party are
  distinct identities. Multi-model, multi-worker, and multi-node execution do
  not prove multi-party federation.
- A receipt proves only the declared boundary fact it binds. Evidence,
  verification, acceptance, adjudication, and settlement are distinct stages.
- Local/domain policy and the applicable authority provider authorize effects;
  the daemon admits, enforces, executes, receipts, and fails closed. Do not say
  that the daemon grants authority.

Conflict rule:

1. Prefer `docs/architecture/` over plans, specs, ignored local master guides,
   prompt packs, reverse-engineering notes, and evidence when architecture
   direction conflicts.
2. If two architecture files disagree, prefer the newer aligned direction. The
   current cross-owner defaults are maintained in
   [`current-canon-defaults.md`](./current-canon-defaults.md). Keep this map
   focused on edit-first ownership.
3. Record resolved contradictions only when the decision history is needed for
   future maintainers; do not keep obsolete variants as parallel doctrine.

## Decision History Policy

Accepted architecture decision history belongs in
[`docs/decisions`](../../decisions/README.md). Do not recreate contradiction
logs as parallel doctrine; port durable rationale into ADRs and keep current
architecture prose clean.

Older plans may keep historical terms such as `adaptive work graph`, former artifact names, or
pre-split capability language only when they are clearly describing decision history.
New canonical architecture must use:

```text
adaptive_work_graph for public delegated execution strategy
prim:* for primitive execution capabilities
scope:* for wallet/provider authority scopes
grant:// or authority_grant_id for authority grants/leases
projection/cache/checkpoint for non-canonical client state
```
