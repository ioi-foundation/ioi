## Why this file exists

The operational-depth atlas,
`apps/hypervisor/application-operational-depth.json`, is a point-in-time audit
of the application plane. This file maps each canonical object to its doctrine
owner, durable target, exact declared repository anchors and roles, required
authority crossing, and application projection. It does not describe delivery
progress, rank work, activate a cut, or sequence a crossing; the sole M0-M14
guide sequences work through admitted work-item records.

Reading rules:

- **Object-anchor coverage is not cut status.** `none` means that the manifest
  declares no `implementation` anchor for the canonical object, even when it
  names an adjacent `precedent`; `partial` means that it declares at least one
  exact `implementation` path. This bounded path-presence observation cannot
  activate, block, or verify a work item or stage. Cut status, retained exit
  evidence, and nonclaims live only in the linked work-item records and the
  derived machine-local `program-state.json` projection.
- **Authority crossing** names the daemon/Agentgres/wallet contract relation
  required for governed truth. It is neither a UI milestone nor an instruction
  to perform that work.
- **Owning application projection** names the product surface that will RENDER
  the object. Projections never own the truth; the daemon/Agentgres contracts
  in the row do.

Machine verification is read-only and runs with
`npm run check:canon-to-code-delta`. The manifest below gives every table row a
stable identity, keeps non-status object-anchor coverage inside the closed
`none | partial` taxonomy, and explicitly distinguishes declared
`implementation` anchors from adjacent `precedent` anchors. Only manifest
paths are machine claims about repository path presence; prose cannot promote
a path, work item, cut, or stage into runtime truth.
